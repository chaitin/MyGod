# 隐私与权限模型

日期：2026-06-29

## 核心原则

AI 助手不能把权限理解成简单的“能看 / 不能看”。对于企业敏感数据，尤其是 HR 薪资数据，产品需要把权限拆成几个独立控制：

- 谁能读取原始数据。
- 助手能否把数据用于内部推理。
- 助手能否把数据透露给另一个人。
- 助手能否基于数据执行动作。
- 这个动作是否需要人工审批。

这意味着：助手可以知道某些信息用于内部判断，但不一定允许把它说出口。

## HR 薪资示例

薪资信息对不同人群应有不同保密级别。

| 人群 | 原始薪资可见性 | 助手可否用于推理 | 助手可否披露 | 典型审批 |
| --- | --- | --- | --- | --- |
| 员工本人 | 只能看自己的薪资 | 可以 | 只能披露本人薪资 | 通常不需要 |
| 直属经理 | 根据公司政策，可能看到团队薪资区间或授权薪酬上下文 | 可以，在团队范围内 | 受限披露，必须符合政策 | 精确数字通常需要审批 |
| HRBP | 可看负责范围内相关员工 | 可以 | 只能向授权对象披露 | 取决于动作 |
| Payroll / Finance | 可看发薪所需数据 | 可以 | 仅限发薪运营上下文 | 通常由流程控制 |
| 部门负责人 | 可看聚合预算或授权区间 | 可以 | 默认只披露聚合或授权数据 | 个人明细通常需要审批 |
| 同事 | 不可见 | 不可用于推理，除非是公开薪酬政策或岗位区间 | 不可披露 | 不允许 |
| 外部招聘方 / 供应商 | 不可见 | 不可用于推理 | 不可披露 | 不允许 |
| 项目频道里的团队 Agent | 默认不可见 | 只有明确授权且目的受限时才可使用 | 不可披露原始薪资 | 需要审批 |

## 数据分级

系统应该做到字段级分类，而不是只对整份文档分类。

薪资相关字段示例：

- `salary.current_amount`：高度敏感。
- `salary.history`：高度敏感。
- `salary.band`：敏感，部分场景可对经理可见。
- `salary.budget_range`：敏感，可能允许聚合可见。
- `bonus.target`：高度敏感。
- `promotion.recommendation`：仅 HR / 经理受限上下文。
- `compensation.policy`：可能公司可见或经理可见。

每个字段都应有元数据：

- 敏感级别。
- 数据主体。
- 所属部门。
- 允许访问的人群。
- 允许使用目的。
- 允许动作。
- 保留期限。
- 是否允许进入 AI 记忆。
- 是否允许在出站回复中引用。

## 目的绑定访问

访问权限不能只看角色，还要看目的。

例如 HR 可以为了以下目的访问薪资数据：

- 工资纠错。
- 薪酬评审。
- 晋升规划。
- 员工自助咨询。
- 预算规划。

同一个 HR 不应该把薪资数据用于无关项目讨论、普通闲聊回复或泛化绩效总结，除非政策明确允许。

助手每次访问敏感数据时，都应绑定一个目的：

```text
用户：HRBP
数据：employee salary.current_amount
目的：年度薪酬评审
允许：读取、向授权审批人总结
不允许：在团队群披露、写入普通项目总结
```

## 读取 / 推理 / 披露 / 行动四层权限

产品应把权限拆成四层：

1. 读取：助手能否检索原始数据。
2. 推理：助手能否把数据用于内部判断。
3. 披露：助手能否向某个人暴露原始或派生信息。
4. 行动：助手能否修改数据、发送消息、创建审批或触发 workflow。

示例：

员工问自己的经理：

> 为什么 Alice 涨薪比我高？

经理的助手可能拥有部分薪酬上下文，但不应该披露 Alice 的薪资或个人绩效细节。

合规回复可以是：

> 我不能分享其他员工的薪酬细节。我可以帮你基于你自己的薪酬区间、你可见的绩效周期记录和公司薪酬政策，准备一次 HR 沟通。

此时：

- Read：在经理范围内也许允许。
- Reason：可以在政策范围内用于判断。
- Reveal：不能披露 Alice 的私人薪资。
- Act：可以草拟合规回复或安排 HR 沟通。

## 面向接收者的脱敏输出

每条 AI 生成的出站消息都应该按接收者过滤。

同一份底层数据，对不同人应该生成不同输出：

### 给员工本人

> 你的当前薪酬处于该级别批准区间内。我可以帮你准备下一次薪酬评审时要问 HR 的问题。

### 给经理

> 该员工接近当前岗位薪酬区间中位数。下一次评审可以结合绩效等级、保留风险和预算情况考虑。

### 给 HR

> 该员工当前为目标中位数的 92%。上次涨薪为 4%。经理备注显示存在保留风险。

### 给同事

> 我不能讨论其他员工的薪酬信息。

同一数据产生不同输出，是因为每个接收者的权限不同。

## 助手默认行为

处理敏感数据时，助手默认应遵守：

- 除非政策明确允许，不披露精确薪资数字。
- 优先使用聚合、区间或政策级解释。
- 涉及薪资、HR、法务、财务、绩效时，优先草拟，不直接发送。
- 接收者或目的不明确时，必须请求确认。
- 无法判断授权时，升级给 HR 或管理员。
- 每一次敏感访问和披露尝试都要留下清晰审计记录。

## 策略规则示例

产品应支持类似策略：

```text
如果 data.category = salary
并且 recipient.role 不在 [self, assigned_hrbp, payroll, authorized_manager]
则禁止披露原始数据。

如果 action = send_message
并且 message.contains_sensitive_salary_data = true
则需要人工审批。

如果 recipient = employee_self
并且 data.subject = recipient
则允许在批准模板内自助披露。

如果 output.audience = team_channel
则只允许披露聚合薪酬预算，不允许披露个人薪酬。
```

## 助手记忆规则

薪资数据不能自动进入普通助手记忆。

推荐记忆模型：

- 个人记忆：用户偏好和日常工作上下文。
- 工作记忆：任务、决策、项目、承诺。
- 敏感记忆：HR、薪资、法务、财务、健康、安全、客户机密。

敏感记忆应该：

- 绑定目的。
- 有时间限制。
- 不参与普通召回。
- 不进入宽泛总结。
- 未授权时从导出中脱敏。
- 每次访问都记录审计。

## 任务边界规则

薪资相关任务卡应包含：

- 数据主体员工。
- 请求人。
- 授权参与者。
- 允许使用的数据字段。
- 允许输出内容。
- 必要审批。
- 到期或复核时间。
- 最终动作类型：草拟、发送、更新 payroll、创建审批、安排会议。

示例：

```text
任务：草拟薪资调整问题回复
主体：员工 A
请求人：员工 A
处理者：HR 助手
允许数据：员工 A 的薪酬区间、公司薪酬政策、评审时间线
禁止数据：其他员工薪资、仅经理可见的内部校准备注
输出模式：仅草稿
审批：发送前需要 HR 审核
```

## 审计与用户信任

用户应该能检查：

- 助手访问了哪些敏感数据。
- 为什么访问这些数据。
- 哪条策略允许访问。
- 回复中包含了什么、脱敏了什么。
- 最终动作是否经过人工批准。

对于敏感 workflow，助手应展示“隐私推理摘要”：

> 我使用了该员工自己的薪酬区间和公开薪酬政策。没有使用或披露其他员工的薪资数据。该回复发送前需要 HR 审批。

## 产品含义

对我们的产品来说，隐私不是合规附加项，而是用户能否相信 AI 助手代表个人或组织工作的核心 UX。

助理应该显得能干，但有边界：

- 能处理常规工作。
- 能智能使用上下文。
- 知道哪些信息是私密的。
- 能解释为什么拒绝或升级。
- 不会把敏感上下文悄悄泄漏到错误房间。
