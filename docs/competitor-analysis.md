# 竞品分析：Kylon、Bloome，以及 Teamily 边界参考

日期：2026-06-29

本分析基于 2026-06-29 可公开访问的网站、文档、应用商店页面、新闻稿和公司内容。

## 执行摘要

Kylon 和 Bloome 是更接近我们产品 thesis 的参考对象。Teamily 只适合作为边界参考，因为它明显偏社交网络：

- Kylon 更偏企业业务执行，定位是让 AI Agent 团队运行公司工作，围绕频道、共享上下文、结构化数据和外部工具展开。
- Bloome 更聚焦 Agent-first group chat，强调人和多个 AI Agent 在同一个聊天空间中工作，并带 Agent marketplace、skill sharing 和 coding agent 集成。
- Teamily 是更宏大的 human + AI social network。它验证了人和 Agent 共存的叙事，但社交图谱、消费社区和泛社交方向使它不适合作为企业 IM 的主要对标。

对我们的结论：方向已被验证，但“AI agents in chat”已经是一个活跃赛道。我们需要更尖锐的切口：不是泛 AI 群聊，而是 assistant-first work layer。每个用户有助理，助理跨邮件、IM、日历、任务和企业系统分流、处理、执行、升级，并具备清晰的身份、权限、审计和隐私边界。

## 对比表

| 产品 | 核心定位 | 最强能力 | 主要优势 | 相对我们的缺口 |
| --- | --- | --- | --- | --- |
| Kylon | 运行企业业务的 AI Agent 团队 | Agent 加入频道、拥有任务、使用工具、更新资源 | 业务执行、结构化数据、集成、权限 | 更像 Agent workspace，未必拥有个人日常消息入口 |
| Bloome | 多 Agent 协作 IM | 人和 Agent 共享群聊、线程、marketplace、skills | 产品焦点清晰，Agent 原生协作机制强 | 仍偏 @mention / chat 驱动，跨渠道 intake 不突出 |
| Teamily | 边界参考：human-AI social network / personal and social AI Agent OS | 长期记忆、多用户多 Agent、社交图谱 | 叙事大、消费/社交分发、完整 AI 网络想象 | 对我们的企业 work-intake thesis 来说太社交化 |

## Teamily：边界参考，不是核心对标

### 它是什么

Teamily 把自己描述为 human-AI social network 和 AI 原生即时通讯产品。人和 AI Agent 可以在同一对话中共存。它的 thesis 是 group intelligence：AI 不只增强个人，也让群体一起思考、创作和建设。

但它不再作为我们的核心对标。它的产品气质太社交化：朋友、家庭、社区、个人 Agent、内容化互动和社交图谱是它的重要方向。我们的目标是企业工作入口、assistant-first 路由、Agent 执行、权限、审计和责任归属。

### 公开资料显示的产品机制

- AI 原生 IM。
- personal and social AI Agent OS。
- 面向朋友、家庭、社区、同事和企业的人机社交网络。
- 每个人可以有一个 AI Agent 团队，而不是只有一个助手。
- 多用户、多 Agent 群聊。
- universal memory：跨群、跨会话、跨 Agent 的持久记忆。
- Agent social network：Agent 可以被提及、被分配任务、回答问题并主动贡献。
- 自定义个人 Agent：通过自然语言创建 Agent，并连接 Gmail、X、Slack、GitHub 等账号。
- 主动 AI：Agent 可以识别意图、总结、推荐并发起任务执行。
- 多模态输入和偏消费社交的内容/发现形态。

### 优势

Teamily 最强的是野心和叙事。它不是“Slack 加 AI”，而是试图重建一个人和 AI 共同生活、共同工作的网络层。如果长期记忆、社交图谱和 Agent 关系链跑通，壁垒会来自数据、关系和记忆的复合网络效应。

### 局限与风险

它的问题也来自范围太宽。个人社交、家庭、社区、企业工作是完全不同的增长模型和信任模型。企业客户关心权限、审计、隔离、合规和数据归属；消费用户关心好玩、易用、隐私和关系密度。两者同时做，产品焦点很难收敛。

### 对我们的启发

- 可借鉴：长期记忆、Agent 身份、多 Agent 共存。
- 不应借鉴：社交 feed、朋友/家庭/社区场景、泛内容发现、消费社交增长模型。
- 结论：Teamily 是边界参考和反例提醒。我们的产品必须更聚焦工作流、权限、审计、责任归属和企业 ROI。

## Kylon

### 它是什么

Kylon 定位为“the AI agent team that runs your entire business”。它是明确的 B2B 产品，核心问题是：公司的上下文散落在 GPT、Codex、Claude、Slack、Gmail、Notion、GitHub、Figma 等工具中，Agent 无法共享上下文、持续执行和跨系统推进工作。

Kylon 的方向是把 Agent、频道、任务、文件、表格、工作流、工具和 API 放进一个共享 workspace。

### 产品机制

- Agent 作为 workspace 成员存在。
- 基于频道的团队聊天和 Agent 协作。
- 共享公司上下文。
- 对话可以转成任务、报告、campaign、follow-up、PR 和 workflow。
- 公司级记忆：一个频道中的反馈和信号可以触发另一个频道中的工作。
- workspace 内有结构化数据：apps、tables、workflows、files、resources，Agent 可以读写。
- 大量集成：GitHub、Slack、Gmail、Salesforce、HubSpot、Linear、Jira、Figma、Stripe、Zoom、Notion、Google Sheets、Calendar、Asana、Shopify、Zendesk、Teams、Outlook、LinkedIn、Dropbox、ClickUp、Airtable、Confluence、Sentry、PostHog 等。
- 信任控制：scoped permissions、human-in-the-loop、RBAC、审计/记录、不用客户 workspace 数据训练、GDPR、SOC 2 Type II in progress。
- 开发者接口：CLI gateway、workspace CLI、proxy API、tool execution API、model proxy API，以及按 Agent/workspace scoped 的 API key。
- 本地 Agent runtime 支持：文档提到 Codex CLI、Claude Code 和 custom runtime 可通过 gateway 连接。

### 优势

Kylon 最大优势是聚焦业务执行，而不是只做对话。它理解企业 Agent 的关键不是“会聊天”，而是能读取上下文、调用工具、修改系统、创建任务、更新数据并留下审计记录。

它的集成层和开发者接口也很关键。如果企业已经使用 Codex CLI、Claude Code、内部脚本、API 和业务系统，Kylon 可以成为 Agent runtime 和企业工具之间的控制层。

### 局限与风险

- 产品仍早期，真实可靠性和企业落地能力需要验证。
- 集成越多，权限、异常处理、速率限制、客户支持和合规成本越高。
- 它更像 Agent workspace / execution layer，不一定自然拥有每个普通员工的日常消息入口。
- 如果用户每天仍在飞书、钉钉、Slack、微信、Gmail、Outlook 中工作，Kylon 需要证明自己能成为入口，而不是多一个工作台。

### 对我们的启发

- Agent 必须能读写工作对象，不能只聊天。
- 权限、审计、工具调用、人类审批是企业 AI IM 的核心。
- 我们若做 assistant-first work intake，要比 Kylon 更重视“人的日常消息请求如何先进入助理”。

## Bloome

### 它是什么

Bloome 是一个人和多个 AI Agent 在同一聊天空间中工作的 IM 平台。它非常接近我们最早提出的痛点：Agent 不应该只是侧边栏或被动机器人，而应该像成员一样进入群聊、上下文和任务。

### 产品机制

- Agent 是一等聊天成员，有 profile 和成员列表存在感。
- 人和 Agent 共享 group chat、DM、reply、thread 和 @mention。
- 多个 Agent 可以在同一对话中协作、委托、共享上下文和并行工作。
- 每个账号都有 personal agent。
- personal agent 可以在 sandbox 中运行代码、读写和编辑文件。
- Cloud Agent beta：Agent 可以运行在 E2B cloud sandbox 中，即使本地设备关闭也能继续工作。
- 支持外部 coding agents：Claude Code、Codex、Gemini CLI、OpenCode 等。
- Agent marketplace：公开 Agent 可浏览、克隆、定制、评分、发布和加入聊天。
- Skill marketplace：SKILL.md bundle 可安装到 Agent、作为 chat card 分享、从 URL 或 zip 安装，并在 Agent 之间复用。
- Agent Collaboration Protocol：Bloome 公开讨论了共享任务状态、新鲜环境感知、安全输出边界等多 Agent 协作问题。
- 多人记忆设计：其博客明确讨论为什么单人记忆模型在多人多 Agent 房间中会失效。

### 优势

Bloome 的优势是焦点清楚：它就是 agent-first group chat。相比 Kylon，它更像 IM；相比传统 Slack bot，它更重视 Agent 成员身份、多 Agent 协作、技能复用和开发者 Agent 接入。

它对 coding agents 的支持是一个很强早期切口。工程团队已经习惯让 Codex、Claude Code 等工具做实际工作，把这些 Agent 带入群聊和任务上下文非常自然。

### 局限与风险

- 公开产品形态仍可能过于 chat-centric。
- 很多交互仍围绕 @mention、群聊和 thread，未必解决“助理先处理，处理不了再找本人”的跨渠道入口问题。
- 外部邮件、IM、日历 intake 不是其最突出定位。
- 企业治理能力比 Kylon 暴露得少。
- Marketplace 会带来质量、信任、安全和维护问题。

### 对我们的启发

- “Agent 作为群成员”已经有直接竞品。
- 多 Agent 同处一个公开房间需要协议级协调，不能只增加上下文。
- Agent marketplace 和 skill marketplace 可以成为分发机制。
- Developer agent 是强早期用户切口。
- 我们若要差异化，应从 assistant-first intake 切入，而不是只做更好的多 Agent 群聊。

## 战略含义

### 1. “Agent 进群聊”已经是明确赛道

Bloome 直接围绕人和 Agent 在同一群聊中协作，Kylon 也有 Agent 加入频道并拥有任务。这个方向已被验证，但也已经拥挤。

产品不能只说：

> Agent 可以加入聊天。

更锋利的承诺应该是：

> 工作先到你的助理。助理跨所有渠道处理、路由、跟进和升级。

### 2. 更大的差异化可能在 intake，而不是 chat

许多产品都在做 chat，但较少产品把助理作为默认工作入口，统一处理：

- 邮件。
- 现有 IM。
- 日历。
- 文档。
- ticket。
- 任务系统。
- CRM。
- 内部工具。

这支持我们的方向：assistant first，human second。

### 3. 企业信任不是功能，而是产品核心

如果 Agent 可以接收消息、代表用户行动、主动发起对话并连接外部工具，那么治理就是产品核心：

- Agent 能代表谁说话？
- 它能读什么？
- 它能写什么？
- 哪些动作需要批准？
- 哪些记忆是个人、团队、公司或禁止访问？
- 动作如何归因？
- 人如何覆盖或审计？

Kylon 是当前公开资料中更值得参考的控制面。

### 4. 记忆必须围绕人、房间和权限设计

Bloome 强调多人记忆；Teamily 虽然偏社交，但也强调全局记忆。对我们来说，记忆层应是：

- 对象感知：每条记忆知道关于谁、什么事。
- 作用域感知：个人、房间、项目、组织、外部联系人。
- 权限感知：Agent 只有在授权时才能检索。
- 目的感知：用于草拟回复的记忆，不等于可用于管理报告的记忆。

### 5. 更好的初始切口

推荐第一切口：

> 团队 AI 工作 inbox：连接邮件和现有 IM，让每个用户有助理；助理处理常规消息和任务，并允许团队 Agent 在共享话题中以明确权限和审计参与工作。

这个方向避免和 Bloome 正面竞争“纯 AI IM”，也避免和 Kylon 正面竞争“Agent 运行整个公司”。

### 6. 可能定位

较完整表达：

> AI-native work router for teams。每个人有一个助理，负责跨邮件、IM、日历和工具处理 incoming work。Agent 是真实 workspace 成员，拥有权限、记忆、后台任务和审计轨迹。

更短表达：

> 让 AI 助手先处理工作，再把真正需要你的部分交给你。

## 下一步问题

1. 我们是做新的 IM，还是先做跨现有 IM/email 的 assistant-first overlay？
2. 初始用户是个人、小型 AI 原生团队，还是企业团队？
3. 第一优先接入渠道是什么：email、Slack/Discord、飞书/钉钉/企业微信，还是通过 adapter 同时支持？
4. v1 允许助理做哪些动作：总结、草拟、回复、排期、建任务、跟进、调用工具？
5. 初始信任模型是什么：所有出站动作人工确认、按规则确认，还是按任务类型配置自治等级？
